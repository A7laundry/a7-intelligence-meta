## Contexto

O `MetaAdsClient._request()` faz uma única tentativa por chamada. Se a API retornar erro transiente (429 Rate Limit, 500 Server Error, timeout de rede), o sistema falha imediatamente. A Meta Marketing API tem rate limits documentados e recomenda implementar retry com backoff.

## Acceptance Criteria

- [ ] Erros transientes (429, 500, 502, 503, 504) são retentados automaticamente
- [ ] Máximo de 3 tentativas com exponential backoff (1s, 2s, 4s)
- [ ] Erros 4xx (exceto 429) não são retentados (erro do cliente)
- [ ] Timeout de requisição configurável (default: 30s)
- [ ] Log de cada retry com nível WARNING
- [ ] Após todas as tentativas, erro original é propagado

## Tarefas

- [x] Criar decorator/helper `retry_with_backoff` reutilizável
- [x] Aplicar retry no `MetaAdsClient._request()`
- [x] Adicionar timeout em todas as chamadas `requests.get/post/delete`
- [x] Aplicar retry no `GoogleAdsApiClient._query()` (erros transientes do gRPC)
- [x] Logar tentativas de retry

## Notas Técnicas

```python
import time

def retry_with_backoff(func, max_retries=3, base_delay=1):
    for attempt in range(max_retries + 1):
        try:
            return func()
        except requests.exceptions.HTTPError as e:
            if e.response and e.response.status_code in (429, 500, 502, 503, 504):
                if attempt < max_retries:
                    delay = base_delay * (2 ** attempt)
                    logging.warning(f"Retry {attempt+1}/{max_retries} após {delay}s")
                    time.sleep(delay)
                    continue
            raise
```
